

AgentWatch
Universal AI Agent Notification & Interrupt System
Master Architecture Plan  ·  v3.0

🎯  What this document is
A complete system architecture plan for AgentWatch — written to be fed directly to AI coding
agents (Antigravity, Gemini, Claude) as build prompts. Each section ends with ready-to-use
prompts. Read it fully before starting any phase.


 
1. The Problem We Are Solving
AI agents are becoming the primary way developers get work done. You give an agent a task — build a feature, refactor a module, research a topic — and then you go do something else while it works. The problem is simple and painful:

😤  The Pain
The agent gets stuck, needs a decision, hits a permission gate, or finishes —
and you have absolutely no idea because you switched to another app.
You end up checking manually, missing it for hours, or babysitting the terminal.
Every AI tool traps its notifications inside itself. There is no universal layer.

AgentWatch solves this by sitting outside every tool, watching passively, and delivering a native OS-level notification the moment anything needs your attention — wherever you are, whatever you are doing.

2. What AgentWatch Is
AgentWatch is a lightweight, local-first background daemon for macOS (and later Windows and Linux) that:

	•	Watches AI agents running anywhere — browser tabs, Mac desktop apps, CLI terminals, IDEs
	•	Reads the full message content AND available UI buttons from whatever the agent outputs
	•	Uses a local offline ML classifier to understand the message and decide what type of event it is
	•	Mirrors the agent's own buttons into a native macOS OS-level notification popup
	•	Lets you respond, approve, deny, or reply directly from the notification without switching apps
	•	Tracks the full history of every notification locally for future reference
	•	Works with zero cooperation from the tool being watched — fully passive observation

🔑  Key Principles
Zero Integration  —  No API keys, no tool cooperation, no wrappers to remember to type.
Fully Local  —  Classifier runs offline. No data leaves your machine. Ever.
Mirror, Don't Invent  —  Buttons in the popup mirror what the tool shows. We don't guess.
Native OS  —  Popups pierce through fullscreen apps using real macOS Cocoa APIs.
Extensible  —  Adding a new tool is a config change, not a code change.

3.1 The Classifier — What It Reads
The classifier does NOT just look at keywords. It reads three signals together to make a decision:

Signal
What It Is
Example
Full message text
The complete text the agent output
"Do you want me to delete the auth module and replace it with JWT?"
Buttons present
All UI buttons visible in the agent interface
["Proceed", "Cancel", "Show Plan"]
UI context
Any label, heading, or status text nearby
"Action Required" or "Task Complete"

From these three signals combined, the classifier outputs:
	•	event_type — what kind of event this is (see Section 4)
	•	confidence — how sure the model is (0.0 to 1.0)
	•	suggested_buttons — which buttons to show in the OS popup
	•	truncated_preview — last 15-20 words for the OS banner title

⚠️  Priority Fallback Rule
When classifier confidence is below 0.75 between two types, always pick the higher-priority one.
It is always better to over-alert the user than to silently miss something important.
Priority order (highest first): ERROR → BLOCKED → PERMISSION → LIMIT → DECISION → RECOMMEND → COMPLETED

📌  Button Mirroring Rule
For PERMISSION and BLOCKED types, buttons in the OS popup must mirror the agent UI exactly.
The Chrome extension scrapes button text from the DOM and passes it in the payload.
The popup engine receives this array and renders them as actionable OS buttons.
The user clicks a button in the OS popup → that same button is programmatically clicked in the browser.

How the Extension Works:
	•	MutationObserver watches the DOM for new agent output
	•	Waits for agent to finish typing — detects by: Stop button disappearing OR 1500ms debounce with no new text
	•	Scrapes: full message text, all button labels, current tab URL, page title
	•	Packages into JSON payload and sends to Core via WebSocket on localhost:7799
	•	Also listens for WebSocket messages back from Core to auto-click buttons or inject text into input fields
	•	One extension manifest covers all 10 tools via URL pattern matching

🐛  Known Extension Issue to Design Around
The Show button must navigate to the EXACT tab, not just the domain homepage.
Solution: Store the full tab URL + tab ID in every payload.
On Show click: use chrome.tabs.update({ active: true }) with stored tabId, then bring window to front.
Do NOT use window.location or simple URL matching — these lose the specific tab.

Layer 2 — Mac Accessibility API (Native Desktop Apps)
Covers Antigravity Mac app and any other native Mac AI application. Uses macOS AXObserver.

	•	Registers AXObserver callbacks on the target application's UI elements
	•	Watches for AXValueChanged and AXUIElementDestroyed notifications on text areas and buttons
	•	Reads AXValue (text content) and AXTitle (button labels) from accessibility tree
	•	Sends same JSON payload format as Chrome Extension to Core WebSocket
	•	Target apps configured in tools.json — add any Mac app by bundle ID, no code change

🐛  Known Mac App Issue to Design Around
AXObserver requires Accessibility permission in System Preferences → Privacy → Accessibility.
The app must request this on first launch with a clear prompt explaining why.
Without it, all AX calls return silently empty — no error, no crash, just nothing.
Always check AXIsProcessTrustedWithOptions() on startup and show a setup guide if false.

Layer 3 — Shell Plugin (Terminal & CLI Tools)
Covers Ollama, Claude Code CLI, Aider, and any terminal-based agent. One-time .zshrc setup.

	•	A 3-line zsh function wraps command output and pipes it to Core via HTTP POST
	•	User adds one line to .zshrc — done forever, no commands to remember
	•	Captures stdout and stderr from any command the function wraps
	•	PowerShell equivalent for Windows (Phase 5)

Layer 4 — Log File Watcher (IDEs & Desktop Apps)
Covers VS Code with Cline/Copilot, Cursor, and any tool that writes agent output to log files.

	•	Uses Python watchdog library to tail log file paths defined in tools.json
	•	Reads new lines as they appear, sends to classifier pipeline
	•	Add any new tool: one line in tools.json with the log path — no code change

 
6. System Architecture

6.1 The Core Daemon — Threading Model
This is the most critical architectural decision. Getting this wrong causes the popup system to deadlock or crash. The solution is a strict dual-loop model:

🏗️  Dual-Loop Concurrency Model
Thread 1 (Background)  —  Runs asyncio event loop + WebSocket server on port 7799.
                           Handles all network I/O: receives payloads, sends replies.

Thread 2 (Main)        —  Runs PyObjC macOS AppHelper.runEventLoop().
                           Handles all UI: renders popups, handles button clicks.

Bridge A (Net → UI)    —  performSelectorOnMainThread() sends payloads from Thread 1 to Thread 2.
Bridge B (UI → Net)    —  asyncio_loop.call_soon_threadsafe() sends replies from Thread 2 to Thread 1.

CRITICAL: Never call PyObjC UI methods from the background thread. Always use Bridge A.

6.2 The Popup Engine — PyObjC Rules
The native macOS popup layer uses PyObjC to call Cocoa APIs directly. Several strict rules must be followed:

🐛  PyObjC KVC Crash — The Main Known Issue
NSUnknownKeyException crashes occur when setting custom properties on NSUserNotification.
RULE: Only use KVC-compliant properties: title, subtitle, informativeText,
      hasActionButton, actionButtonTitle, hasReplyButton, otherButtonTitle.
RULE: Never set additionalActions via KVC — use _additionalActions with underscore prefix
      only if absolutely required, and test on your exact macOS version first.
SAFE ALTERNATIVE: Use UNUserNotificationCenter (modern API) for macOS 10.14+ —
      it is more stable and supports rich actions without KVC pitfalls.

	•	Use UNUserNotificationCenter (UserNotifications framework) as primary popup API
	•	Fall back to NSUserNotification only for macOS versions below 10.14
	•	Request notification permissions on first launch — check UNAuthorizationStatus before every popup
	•	Smart truncation: extract last 15-20 words of agent message for banner subtitle
	•	Full message stored in history DB — accessible via Show button

6.3 The Show Button — Deep Navigation
The Show button must take the user to the exact tab/window/app where the agent is running. This is a common failure point.

🐛  Show Button Known Issue
Simple AppleScript 'open location URL' takes user to home screen, not to the specific tab.
CORRECT SOLUTION for Chrome tabs:
  1. Store tabId and windowId in every payload from the Chrome extension.
  2. On Show click, send a WebSocket message back to the extension.
  3. Extension calls: chrome.windows.update(windowId, {focused: true})
                  then: chrome.tabs.update(tabId, {active: true})
  This reliably focuses the exact tab even across Desktop Spaces.

CORRECT SOLUTION for Mac desktop apps:
  Use NSRunningApplication.activateWithOptions_(NSApplicationActivateIgnoringOtherApps)

6.4 The Return Pipe — Injecting Replies
When the user types a reply in the DECISION popup, that text must be sent back to the agent.

	•	Core sends reply text via WebSocket to Chrome extension
	•	Extension content.js finds the agent input field using tool-specific selectors
	•	Injects text using InputEvent with 'input' type — NOT direct .value assignment (React ignores it)
	•	Then programmatically clicks the submit/send button
	•	For terminal tools: writes reply to a named pipe or PTY stdin

🐛  Return Pipe Known Issue
Direct element.value = 'text' does not work for React-based UIs (ChatGPT, Claude, etc.).
CORRECT: Use Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, 'value').set.call(el, text)
         then dispatch: new Event('input', { bubbles: true })
         This triggers React's synthetic event system correctly.

6.5 Notification History — Local Storage
Every notification is logged locally for future reference. No cloud. No sync.

	•	SQLite database at ~/.agentwatch/history.db
	•	Schema: id, timestamp, tool, event_type, full_message, buttons, user_response, resolved
	•	Accessible via tray icon → History menu
	•	Useful for reviewing what agents did while you were away

12. Summary

AgentWatch is a universal, passive, local-first notification daemon. It watches AI agents across every tool category — browsers, Mac desktop apps, CLI terminals, IDEs — and delivers interactive OS-level notifications the moment they need you.

The system is designed around three principles:

	•	Mirror, don't invent — buttons in the popup mirror what the agent UI shows. The classifier reads the full message AND button signals to understand context and generate the right response options.
	•	Passive everywhere — zero tool cooperation needed. The extension, accessibility watcher, shell plugin, and log watcher all observe silently.
	•	Known issues solved before they occur — every major failure point from previous build attempts is documented in Section 7 with the correct solution pre-designed.

Start with Phase 0. Build the classifier. Get it above 88% accuracy. Then Phase 1 — ChatGPT end-to-end. Every other tool follows the same pattern.
